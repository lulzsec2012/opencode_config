# Submodule Recursive Commit

Automatically detect changes across Git submodules, compute dependency order, and commit from deepest to shallowest, ending with a single merge commit in the parent that captures all submodule pointer updates.

## Overview

This skill handles the common pain point of managing Git submodule projects where changes need to be committed in dependency order: deepest submodules first, then their parents, finally the root superproject.

## Workflow

### Step 1: Detection & Validation

**Before anything else, run safety checks:**

1. **Conflict detection**: Check if any submodule is in the middle of a merge, rebase, or has unmerged files:
   ```bash
   git submodule foreach --recursive 'if [ -f .git/MERGE_HEAD ] || [ -f .git/rebase-merge ] || [ -f .git/rebase-apply ]; then echo "CONFLICT:$name:in_progress_operation"; fi'
   ```

2. **Untracked file check**: List untracked files in each submodule for awareness:
   ```bash
   git submodule foreach --recursive 'git status --short --untracked-files=all'
   ```

3. **Change detection**: Find which submodules have staged or modified (tracked) changes:
   ```bash
   git status --submodule=recursive --short
   git diff --name-only --cached --recursive
   git diff --name-only --recursive
   ```

Report findings to user. If conflicts are detected, stop and report. If only untracked files exist, confirm with user whether to include those (default: skip untracked).

### Step 2: Build Dependency Tree & Compute Order

Read `.gitmodules` to build the full submodule map, then compute depth for each submodule:

```bash
# Get all submodule paths recursively
git submodule foreach --recursive 'echo "$(git rev-parse --show-cdup || echo "")|$(git rev-parse --show-toplevel)"'

# For depth calculation, parse .gitmodules and count nesting levels
# Root = depth 0
# Direct submodules = depth 1
# Submodules of submodules = depth 2, etc.
```

Sort submodules by depth in descending order (deepest first). This ensures child commits happen before parents update their pointers.

### Step 3: Commit Each Submodule (Deepest First)

For each submodule with changes (in computed order):

```bash
# Enter submodule
git -C "$submodule_path" add -u        # Only tracked files, skip untracked

# Check commit message
git -C "$submodule_path" status --short

# Auto-generate commit message from git status/diff summary
# Use conventional commit format
git -C "$submodule_path" commit -m "<auto-generated message>"

# If nothing to commit, git commit will exit cleanly - skip it
```

**Commit message generation rules:**
- Analyze `git diff --stat` and `git status --short` to categorize changes
- Use conventional commit prefixes: `feat:`, `fix:`, `chore:`, `refactor:`, `docs:`
- Include submodule name in message scope: `feat(core): add user authentication` → `feat(core-lib): add user authentication`
- If user provided a message prefix (via `--message` option), prepend it: `<prefix> - <submodule_name>: <auto-generated>`

### Step 4: Parent Pointer Update (Single Merge Commit)

After all submodules are committed, stage the pointer updates in their respective parent repos, working bottom-up:

```bash
# For each parent repo that has submodule pointer changes
git add <submodule_path_1> <submodule_path_2> ...
git add -u  # any other tracked changes in parent

# Generate summary commit message
git commit -m "chore(submodules): update submodule pointers after recursive commit

Changed submodules:
- submodule_a: <brief summary>
- submodule_b: <brief summary>
..."
```

### Step 5: Final Summary

Print a clean summary showing:
- Which submodules were committed (with commit hashes)
- Which were skipped (no changes or user chose to skip)
- Parent commits made with pointer updates
- Total count of commits across all repos

## Configuration Options

Support these options when user invokes the skill:

| Option | Default | Description |
|---|---|---|
| `--dry-run` | off | Show what would be committed without making changes |
| `--message "<prefix>"` | auto | Use custom message prefix for each submodule commit |
| `--include-untracked` | off | Also stage untracked files (default: only tracked changes) |
| `--force` | off | Skip all confirmation prompts |

## Error Handling

- **No changes detected**: Print "No changes found in submodules or superproject." and exit cleanly.
- **Conflict detected**: Print which submodule has conflicts and abort. Do not proceed.
- **Uninitialized submodule**: Print warning and skip. Do not fail.
- **Detached HEAD in submodule**: Print warning to user. Ask whether to create a branch or skip.
- **Network required operations**: Commit only, never push. If submodule needs fetch for some reason, warn and skip.

## Edge Cases to Handle

1. **Submodules with no remote**: Some submodules may be local-only. Still commit, just don't push.
2. **Mixed state**: Some submodules have staged changes, some have unstaged. Stage everything with `git add -u` before committing.
3. **Empty submodules**: Skip submodules that have no changes after staging.
4. **Deeply nested structures**: Recurse fully regardless of depth. The depth-first ordering handles this naturally.
5. **Removed submodules**: If a submodule path in `.gitmodules` no longer exists, skip with a warning.
